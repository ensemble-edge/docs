## What is Edgit?

Edgit is a CLI tool that extends Git with component-aware functionality. It tracks components (functions, agents, configs) in a registry, uses Git tags for semantic versioning, and provides AI-powered commit message generation.

## Why Use Edgit for Everything?

**Edgit is 100% Git-compatible.** Every `git` command works as an `edgit` command. Under the hood, Edgit wraps Git and adds component versioning on top—there's no fork, no replacement, no conflicts.

<Info>
  **TL;DR**: Replace `git` with `edgit` in your daily workflow. You get all Git functionality PLUS component-level versioning and deployment tracking.
</Info>

### How It Works

Edgit wraps Git commands and passes them through:

```bash
# When you run:
edgit add src/hello.ts

# Edgit executes:
git add src/hello.ts

# Plus: Updates component registry if hello.ts is a registered component
```

**Standard commands** (add, push, pull, checkout, merge) → Pass through to Git
**Enhanced commands** (commit, tag, deploy) → Add component intelligence

### What About Existing Workflows?

**Git aliases still work:**
```bash
# Your ~/.gitconfig aliases work through edgit
edgit co -b feature  # Uses your 'co = checkout' alias
edgit ci -m "message"  # Uses your 'ci = commit' alias
```

**CI/CD integration:**
```yaml
# GitHub Actions
- name: Version component
  run: |
    edgit tag my-function v1.0.${{ github.run_number }}
    edgit push origin --tags
```

**Team adoption:**
```bash
# Optional: Add alias for gradual transition
alias git='edgit'

# Now typing 'git' uses edgit automatically
```

### When to Use Git Directly

You only need `git` in these rare cases:

- Checking Git is installed: `git --version`
- Cloning Edgit itself: `git clone https://github.com/ensemble-edge/edgit.git`
- Debugging Git internals: `git fsck`, `git gc`

**Everything else**: Use `edgit`.

## Core Concepts

<AccordionGroup>
  <Accordion icon="database" title="Component Registry">
    The `.edgit/components.json` file tracks all components in your repository with their metadata, paths, and configuration.

    ```json
    {
      "version": "1.0",
      "components": {
        "functions": {
          "email-processor": {
            "type": "function",
            "path": "src/functions/email.ts",
            "description": "Processes incoming emails"
          }
        },
        "agents": {
          "support-agent": {
            "type": "agent",
            "path": "src/agents/support.ts",
            "description": "Customer support automation"
          }
        }
      }
    }
    ```
  </Accordion>

  <Accordion icon="tag" title="Git Tag Versioning">
    Edgit uses Git tags in the format `<component>-v<version>` for versioning. This approach is Git-native and doesn't require version files.

    ```bash
    # Create version tag
    edgit tag email-processor v1.0.0
    # Creates git tag: email-processor-v1.0.0

    # List versions
    edgit tag list email-processor
    # Output: v1.0.0, v1.0.1, v1.1.0

    # View tag details
    git show email-processor-v1.0.0
    ```
  </Accordion>

  <Accordion icon="brain" title="AI-Powered Commits">
    When `OPENAI_API_KEY` is configured, Edgit analyzes component changes and generates conventional commit messages automatically.

    ```bash
    # AI generates message based on diffs
    edgit commit
    # Output: "feat(email-processor): add retry logic for failed sends"

    # Manual override still works
    edgit commit -m "your custom message"
    ```
  </Accordion>

  <Accordion icon="rocket" title="Deployment Tracking">
    Edgit tracks which component versions are deployed to which environments using Git tag metadata.

    ```bash
    # Mark as deployed
    edgit deploy email-processor v1.0.0 --to production

    # Check status
    edgit status email-processor
    # Output:
    # staging: v0.9.0 (deployed 2025-10-15)
    # production: v1.0.0 (deployed 2025-11-01)
    ```
  </Accordion>

  <Accordion icon="list" title="Component Listing & Discovery">
    Explore your components with multiple output formats and discover unregistered component files in your repository.

    ```bash
    # Visual tree showing all components with deployment indicators
    edgit components list --format tree
    # Output:
    # email-processor (function)
    # ├── v1.0.0 [prod]
    # ├── v0.9.0 [staging]
    # └── v0.1.0
    #
    # support-agent (agent)
    # └── v2.1.0 [prod, staging]

    # Find unregistered component files
    edgit components list --untracked
    # Shows files detected as components but not in registry

    # Scan repository for potential components
    edgit discover scan --type prompt
    # Finds all prompt-like files in repository

    # Analyze specific file
    edgit discover detect src/new-function.ts
    # Shows: type, confidence, suggested name
    ```
  </Accordion>
</AccordionGroup>

## Key Features

### Component Management
- **Registry-based tracking**: All components in one place
- **Multiple types**: Functions, agents, prompts, SQL queries, configs, custom types
- **Metadata support**: Description, dependencies, configuration
- **Path-based detection**: Automatically detect component changes in commits
- **Visual exploration**: Tree, table, JSON, and YAML output formats
- **Discovery tools**: Scan repository for potential components
- **Untracked detection**: Find component files not yet registered
- **Deployment indicators**: See which versions are deployed where

### Versioning
- **Git-native tags**: Use existing Git infrastructure
- **Semantic versioning**: Enforced v{major}.{minor}.{patch} format
- **Component-scoped**: Each component has independent version history
- **No package.json clutter**: Versions are tags, not files

### AI Integration
- **Automatic commit messages**: Conventional commits based on diffs
- **Component-aware**: Understands which components changed
- **Configurable**: Use OpenAI or custom providers
- **Opt-in**: Works with or without AI features

### Deployment Workflows
- **Environment tracking**: Know what's deployed where
- **Version history**: Audit trail of all deployments
- **Git integration**: Leverage existing Git workflows
- **Metadata support**: Add custom deployment information

<Steps>
  <Step title="Install">
    ```bash
    npm install -g @ensemble-edge/edgit
    ```

  </Step>

  <Step title="Initialize">
    ```bash
    cd your-project
    edgit init
    ```

  </Step>

  </Step>

  <Step title="Create Version">
